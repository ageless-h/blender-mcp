## Context

当前仓库已具备 MCP 相关代码与文档，但缺少统一的项目结构规划，尤其是核心服务与 Blender 插件端的边界与目录层级。该变更需要在不破坏现有文档与 OpenSpec 流程的前提下，确定长期可维护的目录结构，并明确协议/传输方案的技术选择原则。

## Goals / Non-Goals

**Goals:**
- 明确 MCP 协议与传输方案的技术选择原则，作为项目结构的基础约束。
- 规划核心服务与插件端的独立目录边界与交互契约。
- 统一源码、文档、测试、示例、脚本的分层结构与命名约定。

**Non-Goals:**
- 本阶段不实现具体 MCP 业务能力与 Blender API 逻辑。
- 不引入未经验证的私有 Blender API 依赖。
- 不决定最终部署形态或打包发布方案。

## Decisions

- **协议选择：采用 MCP 官方协议与官方 SDK 作为核心服务实现基础。**
  - Rationale: 保持与 MCP 生态兼容，减少协议偏差与维护成本。
  - Alternatives: 自定义协议或非官方实现（增加兼容性风险，拒绝）。

- **传输方案：MVP 以 stdio 为默认传输，保留后续 HTTP/WebSocket 扩展位。**
  - Rationale: stdio 与 MCP 生态常见实现兼容，适合本地与开发场景。
  - Alternatives: 直接锁定网络传输（增加运维复杂度，暂不优先）。

- **核心服务与插件端严格分离，插件端独立目录。**
  - Rationale: 便于跨版本维护与安全隔离，避免 Blender 版本耦合核心逻辑。
  - Alternatives: 混合式目录（不利于长期维护）。

- **目录结构以 `src/` 为核心代码入口，文档与规范保持在 `docs/` 与 `openspec/`。**
  - Rationale: 与现有文档驱动流程一致，便于工具与测试集成。
  - Alternatives: 扁平结构（难以扩展）。

## Risks / Trade-offs

- **stdio 作为默认传输可能限制远程场景** -> 通过预留传输适配层降低扩展成本。
- **插件端独立目录增加初期组织成本** -> 通过清晰契约与文档减少协作摩擦。
- **依赖官方 SDK 可能受版本演进影响** -> 通过锁定版本与兼容策略降低风险。

## Migration Plan

- 在新目录结构落地前，先完成结构文档与规范的新增与评审。
- 引入 `src/` 与插件端目录时，保持现有代码位置可追溯，避免一次性大规模搬迁。
- 为旧路径提供过渡说明与替换指南。

## Open Questions
